Extending

You're supposed to hack core

app/AppKernel.php


class AppKernel extends Kernel
{
    /**
     * {@inheritDoc}
     */
    public function __construct($environment, $debug)
    {
        // Set the default timezone to UTC.
        date_default_timezone_set('UTC');

        parent::__construct($environment, $debug);
    }

    public function registerBundles()
    {
        $bundles = array(
            new Symfony\Bundle\FrameworkBundle\FrameworkBundle(),
            new Symfony\Bundle\SecurityBundle\SecurityBundle(),
            new Symfony\Bundle\TwigBundle\TwigBundle(),
        );

        if (in_array($this->getEnvironment(), array('dev', 'test'))) {
            $bundles[] = new Symfony\Bundle\WebProfilerBundle\WebProfilerBundle();
            $bundles[] = new Sensio\Bundle\DistributionBundle\SensioDistributionBundle();
        }

        return $bundles;
    }

    // ...
}
  • Bundle ~ Module
  • Bundles install via Composer, add here manually

Symfony project structure

Symfony project structure

  • This is docroot, not project root.
  • Assets get symlinked in here by CLI commands
  • No executable code save app.php!
  • Feel free to edit app.php/app_dev.php

Symfony project structure

  • From Composer
  • Can't touch this
  • Note, includes Symfony itself!

Symfony project structure

  • cache: Generated PHP
  • config: All those YAML config files
  • Pre-build PHP, edit it
  • Resources, app-global templates etc.
  • Nearly all of this is editable by you!

Symfony project structure

  • Your code. AppBundle ~ Custom.module
  • AppBundle.php ~ .info, no discovery, just hard-code class
  • Allows many bundles, but in practice only have one
  • Resuable bundles belong as stand-alone repos, composer

Same kernel, different patterns

Drupal always uses View event

Symfony discourages it

Example Symfony controller


namespace AppBundle\Controller;

use Sensio\Bundle\FrameworkExtraBundle\Configuration\Route;
use Symfony\Bundle\FrameworkBundle\Controller\Controller;
use Symfony\Component\HttpFoundation\Request;

class DefaultController extends Controller
{
    /**
     * @Route("/", name="homepage")
     */
    public function indexAction(Request $request)
    {
        // replace this example code with whatever you need
        return $this->render('default/index.html.twig', array(
            'base_dir' => realpath($this->container->getParameter('kernel.root_dir').'/..'),
        ));
    }
}

Drupal always wants a render array

  • Symfony controllers always *Action
  • $this->render() ~ drupal_render()
  • Array == params to template
  • render() returns Response object!
  • Can also return render-array like structure and use Annotation, discouraged

Paths and routing

  • $module.routing.yml, or event
  • No path nesting
  • Fixed paths with aliasing
  • routing.[yml|xml|php], or annotation
  • Path nesting
  • Slugs
  • Path nesting: Manually mount a module's paths relative to some base
  • Slugs: path fragment == unique lookup key

Storing data

  • Entities
  • State API
  • Configuration API
  • Key/Value
  • Doctrine
  • Can use other ORMs, but few do
  • Doctrine is DBAL and ORM

Doctrine

  • Stand-alone PHP project
  • Simple objects only
  • 1:1 field->property
  • MongoDB requires ODM add-on

Drupal Entities vs Doctrine

  • Configuration-driven
  • Fields
  • Multi-value
  • Rich data types
  • All custom classes
  • Simple properties
  • No multi-value
  • Simple data types
  • As commonly used
  • Much more rudimentary than Drupal

Doctrine gotchas

  • Primitive data only
  • SQL and Mongo are different APIs
  • Different event listeners from Symfony
  • Very primitive file handling

(capes) - Theme whole page at once, with one Twig template/inheritance. - Services, shallow access. - Always 4 ways of doing everything: YAML, XML, Annotation, PHP (at least) -- "Drupal way" is much stricter than "Symfony way". -- May even vary by 3rd party bundle within a single project. Stick to one for your own code. - Metaprogramming via YAML/Annotations rather than user config. Both metaprogramming. Extending *- "You're supposed to hack core". Doctrine migrations, run yourself. -- Drupal has more tools to help here (auto-detect out of sync DB, clear cache from UI, etc.) - Nested routes in Symfony, not in Drupal. - No URL aliases.
